## Conceptos Clave (TL;DR)

* SMTP es un protocolo cliente-servidor, nativamente en texto claro, utilizado para enviar correos electrónicos a través de redes IP.
* Los puertos por defecto son 25 (texto claro), 587 (recepción con STARTTLS) y 465 (conexión cifrada SSL/TLS).
* La arquitectura general de envío de correos sigue el flujo: Client (MUA) -> Submission Agent (MSA) -> Open Relay (MTA) -> Mail Delivery Agent (MDA) -> Mailbox (POP3/IMAP).
* ESMTP (Extended SMTP) es la variante moderna que soporta autenticación segura y cifrado mediante la inicialización de STARTTLS después del comando EHLO.

  
## Herramientas Clave

* **Telnet**: Se utiliza para inicializar conexiones TCP crudas al puerto SMTP, permitiendo la interacción manual y el envío de comandos directamente al servidor.
* **[Nmap](../../../📂%2008%20Herramientas&Cheatsheets/Nmap.md)**: Automatiza la enumeración del servicio SMTP, la extracción de comandos soportados y la identificación de vulnerabilidades como servidores Open Relay mediante scripts NSE.

  
## Metodología Paso a Paso

1. **Reconocimiento y Enumeración**: Utilizar escaneos para identificar puertos SMTP abiertos y listar los comandos soportados por el servidor para entender las capacidades habilitadas.

2. **Interacción Manual**: Establecer una conexión cruda para interactuar directamente con el servicio. Esto permite confirmar las extensiones del servidor e intentar enumerar usuarios del sistema utilizando comandos específicos.

3. **Comprobación de Open Relay**: Verificar, de manera manual o automatizada, si el servidor permite retransmitir correos hacia dominios externos sin exigir autenticación. Si el servidor es vulnerable, se procede a explotar esta condición.

4. **Explotación (Spoofing/Envío)**: Si se identifica una configuración Open Relay, se puede falsificar la identidad del remitente (mail spoofing) para enviar correos electrónicos engañosos o probar el acceso a sistemas internos.

  
## Configuraciones Inseguras
* **Open Relay Global**: Permitir indiscriminadamente rangos de direcciones IP en la configuración (por ejemplo, en Postfix) genera un entorno vulnerable a Open Relay, permitiendo el envío de spam masivo y suplantación de identidad.
* **Parámetro Vulnerable**: `mynetworks = 0.0.0.0/0` o la inclusión de bloques de red excesivamente amplios.

  
## Cheat Sheet de Comandos
```bash
# Enumerar el servicio SMTP, versión y comandos soportados por el servidor
nmap <TARGET_IP> -sC -sV -p<PORT>

  
# Ejecutar test automatizado con 16 pruebas distintas para detectar un Open Relay
nmap <TARGET_IP> -p<PORT> --script smtp-open-relay -v
 

# Conectarse interactivamente al servidor SMTP objetivo
telnet <TARGET_IP> <PORT>
```

Una vez dentro de la sesión de Telnet interactiva:
```text
# Iniciar sesión e identificar el cliente ante el servidor SMTP
HELO <ATTACKER_DOMAIN>

  
# Variante de HELO para ESMTP. Muestra las características y extensiones soportadas
EHLO <ATTACKER_DOMAIN>

  
# Verificar si un usuario específico existe en el sistema (Enumeración)
VRFY <USER>

  
# Verificar la disponibilidad de un buzón/lista de distribución
EXPN <USER>

  
# Declarar la dirección del remitente del correo electrónico (Útil para spoofing)
MAIL FROM: <SPOOFED_EMAIL>

  
# Declarar la dirección del destinatario del correo electrónico
RCPT TO: <TARGET_EMAIL>

  
# Iniciar la transmisión del cuerpo del correo electrónico
DATA

  
# Formato del cuerpo del correo (después de ejecutar DATA)
From: <SPOOFED_EMAIL>
To: <TARGET_EMAIL>
Subject: <SUBJECT_TEXT>
Date: <DATE_FORMAT>
<MESSAGE_BODY>

.
  
# Abortar la transmisión en curso pero mantener la conexión abierta
RSET
 

# Solicitar respuesta del servidor para evitar desconexión por inactividad
NOOP

  
# Finalizar la sesión SMTP
QUIT
```

  
## "Gotchas" y Troubleshooting

* **Falsos positivos en VRFY**: No confíes ciegamente en herramientas automatizadas de enumeración. Dependiendo de la configuración del administrador, el servidor puede emitir el código de respuesta `252 2.0.0 <USER>` confirmando cualquier nombre de usuario enviado, exista o no en el sistema.

* **Ausencia de acuses de recibo fiables**: El uso de SMTP para enviar correos no devuelve una confirmación de entrega nativa fiable; los errores generalmente se devuelven como mensajes genéricos en inglés y dependen del servidor.

* **Transmisión de texto claro**: A menos que se inicialice STARTTLS o se utilice un puerto inherentemente cifrado (como el TCP 465), todos los datos, incluida la autenticación, viajan sin cifrar.

* **Web Proxies**: En ciertos escenarios restrictivos, puede ser necesario tunelizar la conexión SMTP a través de un proxy web enviando el comando `CONNECT <TARGET_IP>:<PORT> HTTP/1.0` al proxy.

* **Inspección de Encabezados**: El encabezado de un correo (regido por RFC 5322) contiene información valiosa de enrutamiento (tiempo, estaciones de relevo), pero no se requiere para la entrega técnica real a nivel de protocolo.
