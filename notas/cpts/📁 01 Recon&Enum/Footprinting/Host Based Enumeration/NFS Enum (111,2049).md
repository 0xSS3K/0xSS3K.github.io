## Conceptos Clave (TL;DR)

* Permite acceder a sistemas de archivos a través de una red como si fueran locales, utilizado principalmente entre sistemas Linux y Unix.
* Se basa en el protocolo ONC-RPC/SUN-RPC expuesto en los puertos TCP/UDP 111, y utiliza el puerto 2049 para el servicio NFS.
* Las versiones más antiguas (v2, v3) no tienen mecanismos propios de autenticación y confían completamente en las opciones del protocolo RPC y en la sintaxis UNIX UID/GID del cliente.
* El servidor NFS confía en la información de usuario enviada por el cliente, lo que permite suplantar accesos creando usuarios locales con el mismo UID/GID.
* La configuración de recursos compartidos se define y gestiona desde el archivo `/etc/exports`.

### Herramientas Clave

* [Nmap](../../../%F0%9F%93%81%2001_Reconocimiento_y_Enumeracion/Footprinting/Host%20Based%20Enumeration/brain-not-braining/%F0%9F%93%82%2008_Herramientas_y_Cheatsheets/Nmap.md) Útil para el footprinting de puertos clave (111, 2049) y cuenta con scripts NSE específicos (nfs-ls, nfs-showmount, nfs-statfs) para enumerar el servicio.
* **rpcinfo**: Enumera los servicios RPC en ejecución, sus versiones y los puertos que están utilizando.
* **showmount**: Permite visualizar la lista de recursos compartidos (exports) disponibles en el servidor.
* **mount / umount**: Comandos nativos de Linux para montar el recurso compartido en nuestro sistema atacante y posteriormente desmontarlo.

### Metodología Paso a Paso

1. **Fase 1: Enumeración de Servicios y Puertos**

&#x20;  Identificar si los puertos 111 (RPC) y 2049 (NFS) están abiertos e interactuar con RPC para mapear los servicios subyacentes.

2. **Fase 2: Descubrimiento de Recursos Compartidos**

&#x20;  Listar qué directorios está exportando el servidor y a qué redes o IPs se les permite el acceso, usando scripts de nmap o la herramienta showmount.

3. **Fase 3: Montaje Local**

&#x20;  Montar el sistema de archivos NFS remoto en una carpeta local para poder navegar e inspeccionar su contenido (archivos de configuración, claves SSH, respaldos) como si estuvieran en el disco propio.

4. **Fase 4: Análisis de UID/GID y Explotación**

&#x20;  Revisar los permisos y los IDs numéricos (UID/GID) de los archivos. Si hay información sensible pero restringida, crear un usuario en el sistema atacante con ese mismo UID/GID para evadir la restricción.

### Cheat Sheet de Comandos

```bash
# Escaneo de puertos clave (111, 2049) descubriendo versiones (-sV) y ejecutando scripts por defecto (-sC)
sudo nmap <TARGET_IP> -p111,2049 -sV -sC

  
# Ejecución agresiva de todos los scripts NSE de enumeración de NFS para listar carpetas, permisos y estadísticas
sudo nmap --script nfs* <TARGET_IP> -sV -p111,2049

  
# Listar los recursos exportados (NFS shares) disponibles en el objetivo
showmount -e <TARGET_IP>

  
# Crear un directorio local y montar el recurso compartido NFS remoto desactivando el bloqueo (-o nolock) para evitar errores
mkdir <LOCAL_DIR>
sudo mount -t nfs <TARGET_IP>:/<SHARE_PATH> ./<LOCAL_DIR>/ -o nolock

  
# Listar el contenido del montaje mostrando usuarios y grupos mapeados localmente
ls -l ./<LOCAL_DIR>/

  
# Listar el contenido del montaje forzando la visualización numérica de UID y GID (crítico para suplantación)
ls -n ./<LOCAL_DIR>/

  
# Desmontar el recurso de red de forma segura una vez extraída la información
sudo umount ./<LOCAL_DIR>
```

### Configuraciones Inseguras

* **rw**: Concede permisos completos de lectura y escritura al cliente.
* **insecure**: Permite el uso de puertos por encima del 1024, lo que posibilita que usuarios sin privilegios de root (los puertos < 1024 son exclusivos de root) interactúen con el servicio NFS.
* **nohide**: Si hay otro sistema de archivos montado dentro de un directorio exportado, este es expuesto automáticamente en lugar de requerir su propia entrada.
* **no\_root\_squash**: Extremadamente crítico; evita que los archivos creados por el usuario root remoto pierdan sus privilegios, conservando el UID/GID 0.

### "Gotchas" y Troubleshooting

* **Restricción root\_squash**: Si esta opción por defecto está activa, cualquier acceso como `root` desde nuestro cliente se mapeará automáticamente al usuario `anonymous` (UID 0 a UID de anónimo), impidiendo la modificación de archivos protegidos incluso si somos root local.
* **Mapeo Manual de IDs**: El servidor no valida quién eres, solo confía en el UID/GID de tu máquina local. Si descubres un archivo que le pertenece al UID 1005, crea un usuario en tu máquina atacante con `useradd -u 1005 atacante` e interactúa con el recurso desde esa cuenta.
* **Vector de Escalada (SUID)**: Si tienes acceso previo por SSH, puedes aprovechar configuraciones sin `root_squash` (o permisos amplios) para subir un binario malicioso al recurso NFS, asignarle permisos SUID del propietario y luego ejecutarlo desde la sesión SSH para obtener una shell como ese usuario.
* **Evolución del Protocolo**: Ten en cuenta que la versión NFSv4 ya solo utiliza el puerto TCP/UDP 2049, eliminando la necesidad de los portmappers (puerto 111), lo que facilita que el servicio pase a través de firewalls.
