## Conceptos Clave (TL;DR)

* La adaptación ante aplicaciones desconocidas y la revisión manual exhaustiva de los resultados de escaneo son indispensables para identificar vectores de ataque que los escáneres automáticos omiten.
* El vector de compromiso más confiable suele ser la combinación de credenciales débiles o por defecto con el abuso de funcionalidades legítimas de la aplicación (built-in functionality) para lograr la Ejecución Remota de Comandos (RCE).
* Aplicaciones de infraestructura, gestión corporativa y monitoreo (vCenter, Zabbix, Nagios, Wikis) representan objetivos de altísimo valor que pueden otorgar un foothold inicial directo o el compromiso total del dominio.

## Herramientas Clave

* **EyeWitness:** Utilizado para manejar reportes masivos de grandes pruebas de penetración, facilitando la revisión visual rápida y el descubrimiento de aplicaciones olvidadas como instancias de Elasticsearch.
* **Metasploit:** Mencionado para asistir en la explotación y despliegue de archivos maliciosos, específicamente útil para el abuso de servicios Axis2.
* **Nessus:** Escáner de vulnerabilidades referenciado por sus puntos ciegos, ya que frecuentemente no detecta vulnerabilidades críticas en ciertas aplicaciones, como Apache Struts 2 en vCenter.
* **JuicyPotato (o similares):** Herramienta esencial de escalamiento de privilegios local (PrivEsc) en entornos Windows, muy efectiva si se obtiene una shell inicial en un appliance de vCenter basado en Windows.

## Metodología Paso a Paso
### Fase 1: Recolección y Filtrado de Ruido

El objetivo de esta fase no es lanzar exploits inmediatamente, sino ordenar la gran cantidad de datos obtenidos de escaneos previos para identificar paneles administrativos expuestos o repositorios abiertos.
* Extraer y revisar visualmente reportes generados por herramientas (como EyeWitness) buscando aplicaciones corporativas atípicas o instalaciones olvidadas de Tomcat, Elasticsearch, o repositorios Git.
* Buscar repositorios Git ampliamente abiertos para extraer contraseñas o llaves SSH que permitan pivotar o acceder a otros sistemas.

### Fase 2: Fingerprinting y Verificación de Credenciales

Antes de buscar vulnerabilidades complejas, se debe interactuar con los portales de autenticación descubiertos utilizando la información pública del proveedor.
* Identificar con precisión la versión del software detectado (Zabbix, Nagios, WebLogic, vCenter, etc.).
* Probar sistemáticamente credenciales por defecto conocidas para el software identificado en consolas administrativas.

### Fase 3: Abuso de Funcionalidad y Explotación

Una vez autenticado (o si el servicio expone fallas no autenticadas), se procede a abusar de la lógica de la aplicación para ejecutar código en el sistema subyacente.
* **Vía Credenciales:** Abusar de la funcionalidad de carga de archivos legítima de la plataforma para desplegar web shells en formatos específicos de la aplicación (ej. WAR para Websphere, AAR para Axis2).
* **Vía Vulnerabilidades Conocidas:** Explotar CVEs públicos, fallas de deserialización en Java (WebLogic), abuso de APIs (Zabbix) o subida de archivos OVA no autenticada (vCenter).

## "Gotchas" y Troubleshooting

* **Axis2 sobre Tomcat:** Axis2 frecuentemente se encuentra ejecutándose sobre una instalación de Tomcat. Si no es posible obtener RCE a través de Tomcat directamente, se debe intentar el acceso a Axis2 para cargar un archivo `.aar`.
* **Puntos ciegos de escáneres (Nessus):** No confíes ciegamente en los reportes de Nessus frente a instalaciones de vCenter; los escáneres automatizados a menudo no logran detectar vulnerabilidades de Apache Struts 2 presentes en estas plataformas.
* **Buscadores de Intranets (SharePoint/Wikis):** Un error común es ignorar la funcionalidad de búsqueda en portales internos. Buscar términos clave dentro de repositorios documentales frecuentemente expone credenciales válidas en texto plano.
* **vCenter PrivEsc y Riesgo de Dominio:** vCenter puede existir como appliance Linux o Windows. Si obtienes una shell en la variante Windows, la escalada a SYSTEM suele ser trivial usando herramientas como JuicyPotato. En muchos entornos, vCenter se ejecuta nativamente como SYSTEM o, configurado erróneamente, como Administrador de Dominio, significando un compromiso total e inmediato.
* **Fallas recurrentes por plataforma:**
    * **Zabbix / Nagios:** Altamente vulnerables a SQLi, XSS almacenado, inyección de código y divulgación de contraseñas LDAP. En Zabbix, la API es el vector principal para ganar RCE.
    * **WebLogic:** Extremadamente susceptible a fallas de Deserialización Java (190 CVEs registrados), lo que a menudo resulta en RCE sin autenticación.
    * **DotNetNuke (DNN):** CMS en C# propenso a bypass de autenticación, directory traversal y bypass en carga de archivos.
