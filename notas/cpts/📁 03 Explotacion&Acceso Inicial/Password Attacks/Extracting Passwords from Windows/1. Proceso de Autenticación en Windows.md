## Conceptos Clave (TL;DR)

* El proceso de autenticación de Windows involucra múltiples módulos para el inicio de sesión, recuperación de credenciales y verificación.

* LSA (Local Security Authority) y su servicio LSASS actúan como el controlador principal que autentica usuarios, hace cumplir las políticas de seguridad local y genera mensajes de auditoría.

* WinLogon es el proceso responsable de interactuar con el usuario, capturar sus credenciales a través de LogonUI y pasarlas a LSASS para su evaluación.

* La base de datos SAM almacena credenciales locales en sistemas que no están en dominio , mientras que NTDS.dit almacena cuentas y hashes de manera centralizada en un entorno de Active Directory.

* El Administrador de Credenciales (Credential Manager) guarda de forma cifrada credenciales de red, sitios web y aplicaciones de manera individual en el perfil de cada usuario.

  
## Herramientas Clave

* **WinLogon y LogonUI**: Procesos del sistema de confianza responsables de gestionar la interacción del usuario y presentar la interfaz gráfica de inicio de sesión.

* **LSASS (Lsass.exe)**: Servicio de subsistema de autoridad de seguridad local que actúa como guardián y gobierna todo el proceso de autenticación en la memoria.

* **Msv1_0.dll / Kerberos.dll**: Paquetes de autenticación (DLLs) utilizados para realizar verificaciones en inicios de sesión interactivos locales y basados en Kerberos, respectivamente.

* **Syskey.exe**: Característica de Windows introducida para cifrar parcialmente el archivo SAM en el disco como protección.

  
## Metodología Paso a Paso
  
**Fase 1: Interacción del Usuario y Captura**

* WinLogon intercepta la solicitud de inicio de sesión (vía mensajes RPC desde Win32k.sys) e inicia LogonUI para solicitar las credenciales.
* Los proveedores de credenciales (implementados como objetos COM en formato DLL) recopilan la información ingresada por el usuario.

  
**Fase 2: Procesamiento y Verificación**

* WinLogon transfiere las credenciales capturadas al servicio LSASS para autenticar al usuario mediante la función LSALogonUser.
* La función Negotiate de LSA determina el protocolo adecuado a utilizar, seleccionando entre NTLM o Kerberos.

  
**Fase 3: Validación de Base de Datos**

* Si el sistema pertenece a un grupo de trabajo (Workgroup), maneja la base de datos SAM y valida las credenciales de los usuarios a nivel local.
* Si el sistema está unido a un dominio, la validación de las credenciales se reenvía al Controlador de Dominio (DC) para que las verifique contra la base de datos de Active Directory (NTDS.dit).

## Problemas

* Las contraseñas de los usuarios se almacenan como hashes (típicamente LM o NTLM) en el registro, siendo susceptibles a técnicas de descifrado offline (offline cracking) si un atacante obtiene acceso a la base de datos SAM.

## Soluciones

* Implementar protecciones criptográficas en la base de datos SAM para prevenir el acceso no autorizado.

* Activar y utilizar SYSKEY para cifrar la base de datos SAM en el disco, asegurando que los hashes de contraseñas locales estén protegidos con una clave generada por el sistema.

* Limitar estrictamente los permisos a nivel de sistema operativo, ya que visualizar o acceder al archivo SAM requiere de privilegios nivel SYSTEM.


## "Gotchas" y Troubleshooting

* **Ubicación y Montaje de SAM:** El archivo físico se encuentra en la ruta `%SystemRoot%\system32\config\SAM` y se monta en el registro bajo el hive `HKLM\SAM`.

* **Requisito de Privilegios:** Se requiere obligatoriamente acceso de nivel SYSTEM para poder visualizar o acceder directamente al archivo SAM.

* **Ubicación de NTDS.dit:** Se almacena en `%SystemRoot%\ntds.dit` y está presente y sincronizado en todos los Controladores de Dominio, a excepción de los RODC (Read-Only Domain Controllers).

* **Ubicación de LSASS:** El binario responsable de la memoria de autenticación local se ubica legítimamente en `%SystemRoot%\System32\Lsass.exe`.

* **Dependencia de Controladores de Dominio:** El paquete `Ntdsa.dll`, encargado de gestionar la base de datos de Active Directory y procesar consultas LDAP, únicamente se carga en los Controladores de Dominio.

* **Aislamiento de WinLogon:** Cada sesión de inicio de sesión interactiva crea una instancia separada del servicio WinLogon.
